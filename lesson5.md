## 1. Browser Supabase client

Для Realtime-подписки нужен Supabase client, который может работать в браузере.

Добавили файл:

```txt
lib/supabase/browser.ts
```

Он создает типизированный клиент через `createBrowserClient` из `@supabase/ssr`:

```ts
createBrowserClient<Database>(supabaseUrl, supabasePublishableKey)
```

Это важно, потому что авторизация в приложении хранится в cookies через `@supabase/ssr`. Если использовать обычный `createClient`, браузерный Realtime-клиент может работать как anonymous user и не получать события из-за RLS-политик.

Также browser client сделан singleton на уровне модуля. Это нужно, чтобы не создавать несколько `GoTrueClient` в одном браузерном контексте с одним и тем же storage key.

Для браузера используется публичный ключ:

```txt
NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY
```

Важно: `SB_SECRET` нельзя использовать на клиенте, потому что это секретный ключ.

## 2. Realtime-компонент для страницы сообщений

Добавили client component:

```txt
app/messages/messages-realtime.tsx
```

Он:

- подключается к Supabase Realtime;
- создает канал `messages-inserts`;
- слушает событие `INSERT`;
- следит за таблицей `public.messages`;
- при новом сообщении вызывает `router.refresh()`.

Также добавили логирование статуса подписки в browser console:

- `SUBSCRIBED` - подписка подключилась;
- `CHANNEL_ERROR` или `TIMED_OUT` - подписка не работает, нужно проверять Realtime publication, RLS policies или ключи Supabase.

Дополнительно исправили warning React про `useEffect`: dependency array должен всегда иметь одинаковый размер. Для этого `selectedClientId` нормализуется в `clientId`, а effect всегда зависит от:

```ts
[router, clientId]
```

Позже фильтр по `client_id` был временно убран, чтобы проверить сам факт подключения Realtime. Сейчас подписка слушает все `INSERT` события в `public.messages` и обновляет страницу при любом новом сообщении.

Также добавили проверочный запрос из браузера:

```ts
supabase.from("messages").select("id", { count: "exact", head: true })
```

Он помогает понять, может ли browser Supabase client читать таблицу `messages`. Если этот запрос падает, значит Realtime, скорее всего, блокируется RLS/session, а не кодом подписки.

Это позволяет заново получить серверные данные и перерисовать страницу сообщений без ручного обновления браузера.

## 4. Почему migration не нужна в Next-приложении

Чтобы Supabase мог отправлять события Realtime по таблице `messages`, таблица должна быть добавлена в публикацию `supabase_realtime`.

Изначально migration была добавлена в этот Next-проект:

```txt
supabase/migrations/20260505000000_enable_messages_realtime.sql
```

Но это не лучшее место для такой migration, потому что база данных управляется отдельным backend-проектом. Все изменения схемы БД, публикаций, таблиц и SQL migration должны храниться и применяться именно там.

В Next-приложении migration удалена. Здесь остается только frontend-логика:

- создать browser Supabase client;
- подписаться на Realtime-события;
- обновить UI через `router.refresh()`.

SQL, который должен быть применен в backend-проекте:

```sql
alter publication supabase_realtime add table public.messages;
```

Если backend уже включает `messages` в Realtime publication, в Next-проекте ничего дополнительно делать не нужно.

## 7. Непрочитанные сообщения

На backend были добавлены поля в таблицу `messages`:

```txt
read_at
read_by_manager_id
```

В Next-приложении добавили чтение этих полей и логику непрочитанных сообщений.

Как считается счетчик:

- берутся сообщения, у которых `read_at is null`;
- они группируются по `client_id`;
- для каждого клиента считается `unread_count`.

В UI счетчик отображается в левом списке клиентов:

```txt
3 unread
```

Если непрочитанных сообщений нет, бейдж не показывается.

Также добавили правило видимости непрочитанных сообщений по назначению:

- если клиент никому не назначен, его unread-счетчик видят менеджеры;
- если клиент назначен текущему менеджеру, текущий менеджер видит unread-счетчик;
- если клиент назначен другому менеджеру, текущий менеджер не видит unread-счетчик этого клиента.

Для этого `getClients()` теперь получает `currentManagerId` и обнуляет `unread_count`, если клиент назначен другому менеджеру.

## 8. Подсветка новых сообщений и пометка как прочитанных

Новые сообщения в открытом диалоге подсвечиваются, если:

```txt
read_at is null
```

В UI такие сообщения получают:

- компактную темную подсветку;
- заметную вертикальную метку слева;
- остаются видимыми как новые, пока менеджер не начнет отвечать.

Сообщения больше не помечаются прочитанными сразу при открытии диалога. Это нужно, чтобы менеджер успел увидеть, какие именно сообщения были новыми.

Когда менеджер отправляет ответ через `Send`, приложение вызывает server action:

```ts
markClientMessagesAsRead(clientId)
```

Action обновляет все непрочитанные сообщения этого клиента:

```ts
read_at = now()
read_by_manager_id = currentManager.id
```

После этого сообщения считаются прочитанными для всей системы, потому что если один менеджер прочитал сообщения, другим менеджерам они уже не должны отображаться как новые.

Компонент, который раньше автоматически помечал сообщения при открытии диалога, был удален:

```txt
app/messages/mark-messages-read.tsx
```

Теперь логика запускается из:

```txt
app/messages/reply-composer.tsx
```

Почему не помечаем прочитанным при наборе текста:

- менеджер может начать писать и передумать;
- текст может остаться черновиком без ответа клиенту;
- клиентские сообщения должны считаться обработанными только после реального ответа.

После успешного выполнения action вызывается `router.refresh()`, чтобы счетчик unread и подсветка новых сообщений исчезли в интерфейсе.

Важно: текущий `lib/supabase/database.types.ts` еще не обновлен под новые поля `read_at` и `read_by_manager_id`, поэтому временно используются локальные type overrides. После регенерации Supabase types эти обходы можно убрать.

## 9. Отправка ответа менеджера клиенту

Добавили отправку ответа менеджера через Supabase Edge Function:

```txt
manager-reply
```

В Next-приложении добавили server action:

```ts
sendManagerReplyAction(clientId, text)
```

Что делает action:

1. Проверяет, что текст не пустой.
2. Получает текущего авторизованного менеджера.
3. Проверяет, что выбранный клиент назначен именно этому менеджеру.
4. Вызывает Edge Function:

```ts
supabase.functions.invoke("manager-reply", {
  body: {
    client_id: clientId,
    text: replyText,
  },
});
```

Важно: `client_id` - это `clients.id`, а не `telegram_chat_id`.

После успешной отправки:

- черновик ответа очищается;
- текст удаляется из `localStorage`;
- страница обновляется через `router.refresh()`;
- серверный путь `/messages` инвалидируется через `revalidatePath`.

Файлы:

```txt
app/messages/actions.ts
app/messages/reply-composer.tsx
```

## 14. Заготовка страницы базы знаний

Добавили новую страницу:

```txt
/knowledge-base
```

Зачем нужна эта страница:

- в будущем менеджеры смогут заполнять базу знаний;
- база знаний будет состоять из пар `вопрос - ответ`;
- эти материалы можно будет использовать для будущих AI-ответов клиентам;
- сейчас страница сделана как визуальная заготовка без сохранения в базу.

Почему начали именно с визуальной заготовки:

- нам нужно заранее понять, как менеджер будет работать с будущей базой знаний;
- до подключения таблиц Supabase можно быстро проверить структуру интерфейса;
- проще согласовать поля и сценарии сейчас, чем менять уже готовую backend-логику позже;
- эта страница готовит основу для будущих AI-ответов, но пока не влияет на реальные сообщения клиентам.

Что есть в интерфейсе:

- общий header с переходом между `Сообщения` и `База знаний`;
- список заготовок слева;
- строка поиска по названию вопроса;
- при вводе текста список автоматически фильтрует похожие вопросы;
- центральная форма карточки знания;
- поля для вопроса, ответа и категории;
- правый блок проверки качества материала перед использованием AI;
- одна кнопка `Сохранить` как визуальная подготовка будущей логики.

Что специально убрали из интерфейса:

- кнопку `Опубликовать`, потому что пока нет реального статуса публикации;
- статусы карточек (`Черновик`, `На проверке`, `Опубликовано`), чтобы не усложнять заготовку раньше времени;
- поле `Приоритет`, потому что сейчас для базы знаний важнее сам вопрос и корректный ответ;
- поле `Ответственный раздел`, потому что пока нет согласованной логики распределения материалов по отделам.

Текущий рабочий сценарий страницы:

1. Менеджер открывает `База знаний`.
2. Видит список существующих заготовок вопросов.
3. Может начать вводить вопрос в поиск.
4. Список слева сразу показывает похожие вопросы.
5. Менеджер видит карточку знания: вопрос, ответ и категорию.
6. В будущем менеджер сможет сохранить эту карточку в Supabase.

Страница защищена так же, как `/messages`: сначала получаем авторизованного менеджера через Supabase, затем отображаем интерфейс.

Разделение по файлам:

- `app/knowledge-base/page.tsx` остается server component;
- в нем выполняется проверка авторизованного менеджера;
- `app/knowledge-base/knowledge-base-workspace.tsx` является client component;
- в client component вынесены интерактивные части: поиск, ввод текста, будущие действия формы.

Почему так:

- авторизацию и получение менеджера лучше держать на сервере;
- живой поиск требует `useState`, поэтому он должен быть в client component;
- такое разделение сохраняет страницу безопасной и при этом дает интерактивный интерфейс.

Сейчас список вопросов хранится временно прямо в коде:

```ts
const draftArticles = [...]
```

Позже этот массив нужно заменить на данные из Supabase, например из таблицы `knowledge_base`.

Поиск сейчас работает локально:

```ts
article.question.toLowerCase().includes(normalizedQuery)
```

Этого достаточно для заготовки. Когда материалов станет много, поиск лучше перенести на уровень базы данных.

Файлы:

## 15. Автоматическое сообщение клиенту при назначении менеджера

Добавили отправку автоматического сообщения клиенту, когда диалог назначают на менеджера.

Зачем это нужно:

- клиент должен понимать, что его вопрос не потерялся;
- клиент видит, что обращение уже передано конкретному специалисту;
- менеджеру не нужно вручную писать первое сервисное сообщение после назначения;
- в чате остается история: когда и кому был передан вопрос.

Логика:

- если менеджера сняли с клиента (`No manager`), сообщение не отправляется;
- если клиент уже был назначен на этого же менеджера, повторное сообщение не отправляется;
- если клиента назначили на нового менеджера, клиент получает уведомление в Telegram через Edge Function `manager-reply`.

Текст сообщения:

```txt
Здравствуйте! Ваш вопрос уже передан нашему специалисту.

С вами будет работать Имя Фамилия, должность.
Специалист уже получил ваше обращение, внимательно его рассмотрит и в ближайшее время вернется к вам с ответом.
```

Так клиент понимает, что обращение принято, передано нужному специалисту и находится в работе.

Как это работает в коде:

1. В `assignClientAction()` читаем `clientId` и `managerId` из формы назначения.
2. Получаем текущего авторизованного менеджера и его Supabase access token.
3. Читаем предыдущее назначение из `client_assignments`.
4. Сравниваем старого менеджера с новым.
5. Если менеджер не изменился, уведомление не отправляем.
6. Если выбран `No manager`, уведомление не отправляем.
7. Если назначен новый менеджер, загружаем его `name`, `surname`, `position`.
8. Обновляем `client_assignments`.
9. После успешного обновления вызываем Edge Function `manager-reply`.

Почему сначала проверяем предыдущее назначение:

- чтобы не отправлять одинаковые сообщения клиенту при повторном нажатии `Save`;
- чтобы клиент не получал сервисное сообщение, когда менеджер просто сохранил форму без изменений;
- чтобы уведомление отправлялось только при реальной передаче вопроса специалисту.

Почему используем Edge Function `manager-reply`:

- отправка в Telegram уже реализована на backend;
- Next-приложение не должно напрямую знать детали Telegram API;
- Edge Function сама записывает исходящее сообщение в таблицу `messages`;
- после записи Realtime обновляет интерфейс сообщений.

Важно: если Edge Function вернет ошибку, action выбросит ошибку. Назначение к этому моменту уже может быть обновлено в `client_assignments`, но менеджер увидит проблему и сможет повторить действие или отправить сообщение вручную.
